# Architecture Overview

## System Architecture

Pro Grocery is a client-server Flutter application using Firebase as the backend. The architecture follows a layered pattern separating UI, business logic, and data access.

```
┌─────────────────────────────────────────────────┐
│                   UI Layer                       │
│  Views (Screens) ← Riverpod / Provider          │
├─────────────────────────────────────────────────┤
│               Business Logic Layer               │
│  Services (stateless) + Providers (stateful)     │
├─────────────────────────────────────────────────┤
│                Data Layer                        │
│  Firestore │ Hive (local cache) │ REST APIs     │
└─────────────────────────────────────────────────┘
```

### Riverpod Root

`main.dart` wraps the app in `ProviderScope` (Riverpod's root). `LocaleProvider` is registered as a `ChangeNotifierProvider` at the top level alongside Riverpod.

### State Patterns Used

| Pattern | Usage |
|---------|-------|
| `StreamProvider` | Real-time Firestore listeners (auth, flash deals, orders, notifications, wallet) |
| `FutureProvider` | One-shot async data loads (products, categories, coupons, recipes) |
| `FutureProvider.family` | Parameterized queries (product by ID, products by category, search) |
| `StateNotifier` | Complex mutable state with business logic (cart, wishlist, orders, wallet, loyalty, coupons, settings, notifications, recipes, user profile) |
| `StateProvider` | Simple filter/toggle state (product filter) |
| `Provider` | Service instantiation and derived/computed values |

## Routing

**GoRouter 14.8+** manages all navigation with type-safe deep linking.

- Route definitions: `lib/core/routes/app_router.dart`
- **64 routes** covering splash, auth, home, product, cart, checkout, orders, profile, settings, deals, recipes, loyalty, referrals, AI search, bundles
- Auth redirect logic: unauthenticated users redirect to `/loginOrSignup`; authenticated users on auth routes redirect to `/entry_point`
- **32 protected routes** requiring authentication
- Extra data passed via `state.extra` as `Map<String, dynamic>` for routes needing parameters

## Data Flow

```
User Action (UI)
    ↓
Riverpod Provider (state change via StateNotifier/FutureProvider)
    ↓
Service (Firestore/Firebase/API call)
    ↓
Firestore / Hive Cache / External API
    ↓
Service returns data
    ↓
Provider notifies listeners (AsyncValue)
    ↓
UI rebuilds via .when(data:, loading:, error:)
```

## Offline Architecture

```
┌──────────┐     ┌──────────┐     ┌──────────┐
│ Firestore│ ←→  │ Riverpod │ ←→  │   UI     │
└──────────┘     └────┬─────┘     └──────────┘
                      │
                 ┌────┴─────┐
                 │   Hive   │
                 │  Cache   │
                 └──────────┘
```

- **Hive** provides key-value local storage with dedicated boxes: products, cart, categories, recipes, deals, meta
- Auto-caches API responses with configurable TTL (default 30 minutes via `HiveService.isStale()`)
- Falls back to cache when network unavailable
- `HiveService.clearAll()` clears all boxes (called on logout)
- Cache freshness tracked via `setLastSync(key)` timestamp in meta box

## Environment Configuration

Config is loaded via `EnvConfig` (`lib/core/config/env_config.dart`) which wraps `flutter_dotenv`. API endpoint constants live in `lib/core/config/api_endpoints.dart`.

## Key Design Decisions

1. **Riverpod as primary state management:** All providers use Riverpod patterns (`StreamProvider`, `FutureProvider`, `StateNotifier`). Only `LocaleProvider` uses `ChangeNotifier`.
2. **Firestore as primary DB:** No custom backend server; Firebase handles auth, DB, and notifications
3. **Hive for offline:** Dedicated boxes per data type with 30-min TTL via `isStale()`; cleared on logout
4. **GoRouter over Navigator 2.0:** Type-safe routing with 64 routes and auth redirect logic
5. **Service layer is stateless:** All services are helper classes, not state containers
6. **Cloudinary for images:** Decouples image hosting from Firestore (Spark plan compatible)
7. **Dual payment providers:** M-Pesa for East Africa, Flutterwave for broader Africa/global
8. **Environment-based config:** `.env.{APP_ENV}` loaded via `flutter_dotenv` with typed accessors in `EnvConfig`
